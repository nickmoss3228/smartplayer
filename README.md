# История создания "The Infinity Player"

## Введение

Всем привет!

Здесь я расскажу историю создания **"The Infinity Player"** и какие трудности стояли при разработке данного проекта. Поговорим про техническую, методологическую и профильную часть.

---

## 1. Истоки

Все началось с идеи. Я работаю в школе английского языка, и мы заметили одну особенность: при преподавании аспектов языка, таких как грамматика, лексика, аудирование, чтение, разговорный английский и письмо - мы видим, как у студентов всегда отстаёт один аспект - **аудирование**. 

### Проблема

Так как практика всех остальных аспектов происходит на уроке и дома, аудирование - это самый непрактикуемый аспект английского языка. Даже если мы возьмем аудирование на уроке - чтобы понять услышанное - часто приходится переслушивать аудио по 3-4 раза, что занимает много времени, поэтому всё сводится к прослушиванию 1-2 раза. 

За это время, конечно, не всем удаётся понять, что было сказано или что имелось ввиду. И многие остаются недовольными - и сдаются понимать английскую речь. К тому же, многие студенты не практикуют прослушивание английского с помощью видео или аудио материалов, а искать что-то самим или смотреть фильмы на английском - нет сил или времени.

### Решение

Так и родилась идея - воплотить в реальность одну методику, которая была придумана американским учителем. Она заключается в том, чтобы у вас была возможность прослушать аудио полностью, и затем переслушивать отдельные части предложения или предложения целиком - для того, чтобы ваш мозг успел проработать информацию. 

Аудио можно разбить на предложения, и дать возможность студентам слушать предложения, фокусируясь на темпе предложений и их ударении. Технически, данная идея была возможна, и я стал продумывать способы разработки и стек.

Я начал с размышления вариантов технологий и языков, но одно было очевидным - я знал что данный проект должен быть в ключе веб-разработки. Это будет сайт, на котором можно практиковаться, сохранять свой прогресс и пользоваться им как на компьютере - так и на телефоне.

## 3. Player

### Выбор решения для плеера

Мне нужно было найти готовый плеер либо сделать такой самостоятельно. Я попробовал сделать его сам, и у меня вышел очень примитивный вариант, который выглядел плохо и был ограничен в функционале. 

Поэтому, я решил найти уже готовый вариант - но с визуализацией звуковой дорожки - прямо как в Dropbox или Soundcloud. Я думаю, что данное решение помогает визуализировать предложения и видеть перед собой - сколько еще нужно прослушать или сколько уже было пройдено - и в случае с обучением - это самый надежный вариант. 

Выбор пал на **WaveSurfer** - в нём было очень много встроенных функций по кастомизации дорожки, и его стиль можно с легкостью подстроить под себя.

### Проектирование функционала

WaveSurfer был лишь частью визуализации, но не всем смыслом. Мне нужно было продумать, какие функции должен нести в себе этот плеер. Заставить его включать аудио или менять аудио было тривиальной задачей, но внедрить новый функционал уже было трудным заданием. Поэтому, я решил написать псевдо-код про функционал данного проекта.

#### Основные требования к функционалу:

- В плеере должна быть кнопка, которая включает специальный режим работы. В нём, нам являются доступными две кнопки: 
  - одна отвечает за **повторение предложения** внутри определенных границ
  - другая - **переходит к следующей границе**

Для этого, нужны функции старт\стоп, функция включения или выключения данного режима, а также функции которые отрабатывают при нажатии кнопок повтора и перехода к следующей части текста. Причём, каждая функция должна учитывать состояние другой, так как мы отслеживаем прогресс в аудиозаписи.

- Также, было необходимо сделать **проверку знаний** по прослушанному - таким образом, был создан компонент с викториной (`Quiz.tsx`).

## 5. Инициализация WaveSurfer

Я начал с инициализации Wavesurfer: прописал туда параметры, которые должны быть на момент запуска.

### Процесс инициализации

Инициализация происходит внутри хука `useEffect`, где мы сначала проверяем наличие ссылки на аудио, либо присутствие рефа на WaveSurfer. 

**Этапы:**

1. Подчищаем предыдущее использование плеера, вызывая методы для остановки, удаления и обнуления предыдущего референсного значения
2. Меняем состояние play/pause на `false` - чтобы остановить аудио, если мы вышли со страницы, или зашли заново
3. В `wavesurfer.current` - мы вызываем инициализацию, где прописываем все инструкции
4. Создаем переменную `instance` - для слушателей событий

### Слушатели событий

#### `instance.on("ready" => ...)`

Подготавливаем плеер к работе:
- Громкость берем из состояния, если стоит без звука то приравниваем нулю, в остальном случае - достаем из состояния значение громкости
- Отслеживаем и считаем начальное время аудио и конечное
- Вызываем функцию из WaveSurfer под названием `getDuration` - это встроенная функция плеера, которая возвращает длительность аудио в секундах
- Присваиваем это значение в состояние, но при этом в общую длительность нужно отформатировать время в человеческий вид
- Вызываем ещё одну функцию - `formatTime`

**Остальные слушатели** просто меняют состояние из стейт-менеджера. 

### Отслеживание движения указателя

Внутри компонента с инициализацией также есть функция, которая отслеживает движение указателя (`handlePointerMove`) и визуализирует передвижение мышки по аудио треку. Мы ищем с помощью `querySelector` тэг `#hover`, и если это `PointerEvent` - то переопределяем ширину данного компонента - основываясь на позиции курсора.

### Cleanup функция

- Мы подгружаем аудио ближе к концу данного `useEffect`
- Создаем cleanup функцию, в которой мы вызываем функцию `cleanInterval` - если такой имелся
- Удаляем слушатель событий на аудио дорожке - `handlePointerMove`
- Удаляем инициализированную копию плеера и удаляем все события
- Меняем флаг инициализации на `false`

> **Важно:** Данный `useEffect` будет перерисовываться ТОЛЬКО в случае смены аудио.

## 6. Ключевые функции плеера

### Функция `getCurrentSegmentBounds`

Следующая функция, которая используется практически во всех других слушателях событий - это **`getCurrentSegmentBounds`**.

В данной функции главной идеей заключается вычисление маркера, на котором мы находимся сейчас, и маркера который идет после.

**Логика работы:**

- Инициализируем переменные `nextMarker` / `currentMarker`. Они берут значение маркеров из глобального объекта, где содержится вся информация про дорожки
- `nextMarker` также забирает значение оттуда, но + 1
- Далее мы вычисляем **начало** и **конец**:
  - **Начало:** если тип это объект (со всеми маркерами), то отрисуем время данного маркера
  - **Конец:** если тип это объект (со всеми маркерами) и если есть время следующего маркера, то отрисуем именно его

Всю функцию мы оборачиваем в `useCallback`, и перевызываем если меняется маркер, индекс маркера, либо секунды длительности.

> **`currentMarkerIndex`** - это глобальное состояние, чтобы понять на каком маркере мы находимся, он находится там потому что нам нужен доступ к нему всегда и из других функций.

---

### Функция `updateCurrentMarkerIndex`

Очень важной функцией также является **`updateCurrentMarkerIndex`**, которая позволяет определить, на каком предложении находится плеер.

#### Основная проблема и её решение

> Следующая проблема сводила меня с ума долгое время. Я не мог понять, почему маркеры времени не воспроизводятся по очереди регулярно. Иногда они работали исправно, а иногда - воспроизведение начиналось со СЛЕДУЮЩЕГО маркера. Хотя ожидается, что мы перейдем обратно на предыдущий маркер. Ответ заключался в нерегулярности завершения аудио и фреймах.

Оно принимает в себя текущее время воспроизведения. Для начала, мы проверяем инициализацию Sentence Mode и есть ли маркеры предложений.

**Решение:**

Во-первых, мы рассчитываем начало и конец с помощью `getCurrentSegmentBounds()`, и забираем оттуда конец. После чего сравниваем его со временем текущего воспроизведения. И если оно будет больше или равно конечному результату но - 0.1, то тогда мы выходим из функции. 

**Это сделано для того, чтобы индекс не "прыгнул" на следующее предложение прямо перед паузой.**

#### Поиск текущего маркера

Основная часть заключается в том, чтобы найти индекс маркера, в интервале которого находится время текущего воспроизведения.

```javascript
const newIndex = timeMarkers.findIndex((marker, index) => {
  // ... логика внутри
});
```

**Шаг 1: Извлекаем время текущего маркера**

```javascript
const currentMarkerTime = typeof marker === "object" ? marker.time : marker;
```

**Шаг 2: Определяем время следующего маркера**

```javascript
if (index < timeMarkers.length - 1) {
  const nextMarker = timeMarkers[index + 1];
  nextMarkerTime = typeof nextMarker === "object" ? nextMarker.time : nextMarker;
} else {
  nextMarkerTime = durationSeconds || Infinity;
}
```

- Если у нас не последний маркер, то мы берем время следующего маркера
- А если последний - тогда берем длительность аудио или бесконечное число

**Пример:**

```javascript
timeMarkers = [
  { time: 0, text: "Hello" },    // index=0
  { time: 5, text: "World" },    // index=1
  { time: 10, text: "Bye" }      // index=2 (последний)
];

durationSeconds = 15;

// Для index=1:
currentMarkerTime = 5
nextMarkerTime = 10  // из timeMarkers[2]

// Для index=2:
currentMarkerTime = 10
nextMarkerTime = 15  // из durationSeconds
```

**Шаг 3: Проверяем, попали ли мы в интервал**

```javascript
return (
  currentTimeValue >= currentMarkerTime &&
  currentTimeValue < nextMarkerTime
);
```

**Визуализация:**

```
Маркеры:     [0]      [5]      [10]     [15]
Интервалы: [0...5]  [5...10] [10...15]

currentTimeValue = 7.2

7.2 >= 5 ✅  И  7.2 < 10 ✅
= Подходит index=1
```

**Шаг 4: Обновляем состояние**

И в конце - мы обновляем состояние (если два условия верны: маркер найден и индекс изменился).

```javascript
if (newIndex >= 0 && newIndex !== currentMarkerIndex) {
  dispatch(setCurrentMarkerIndex(newIndex));
}
```

**Полная визуализация процесса:**

```
Маркеры предложений:
┌─────┬─────┬──────┬──────┐
│  0s │  5s │ 10s  │ 15s  │
└─────┴─────┴──────┴──────┘
   0     1      2     3 (индексы)

Воспроизведение:
-------▶ currentTime = 7.2s
           
Функция выполняет:
1. Проверяет: 7.2 < (10 - 0.1)? (не близко к концу сегмента, не сработает)
2. Ищет интервал:
   - [0, 5]?   7.2 >= 0 ✅, но 7.2 < 5 ❌
   - [5, 10]?  7.2 >= 5 ✅, и 7.2 < 10 ✅  → НАЙДЕНО!
3. newIndex = 1
4. Если currentMarkerIndex был 0 → dispatch(setCurrentMarkerIndex(1))
```

## 7. RAF-цикл (RequestAnimationFrame)

И, в завершение описания важнейших функций для работы плеера - это `useEffect`, который выполняет полный цикл RAF.

```javascript
useEffect(() => {
  if (!wavesurfer.current || !isInitialized) return;

  const instance = wavesurfer.current;
  let animationFrame: number;

  const updateProgress = () => {
    if (!instance) return;

    const currentTimeValue = instance.getCurrentTime();
    dispatch(setCurrentTime(formatTime(currentTimeValue)));
    updateActiveSubtitle(currentTimeValue);

    // Handle stopping BEFORE updating marker index
    if (isPlayMode && timeMarkers?.length) {
      const { end } = getCurrentSegmentBounds();

      // More precise stopping with smaller threshold
      if (currentTimeValue >= end - 0.02) {
        instance.pause();
        // Ensure we don't update marker index after stopping
        if (isPlaying) {
          animationFrame = requestAnimationFrame(updateProgress);
        }
        return;
      }
    }

    // Update marker index only if we're not stopping
    updateCurrentMarkerIndex(currentTimeValue);

    if (isPlaying) {
      animationFrame = requestAnimationFrame(updateProgress);
    }
  };

  if (isPlaying) {
    animationFrame = requestAnimationFrame(updateProgress);
  }

  return () => {
    if (animationFrame) {
      cancelAnimationFrame(animationFrame);
    }
  };
}, [
  isPlaying,
  isInitialized,
  isPlayMode,
  getCurrentSegmentBounds,
  updateActiveSubtitle,
  updateCurrentMarkerIndex,
  dispatch,
]);
```

### Назначение функции

Данная функция запускает **60 FPS цикл обновления UI** во время воспроизведения:
- Обновляет время
- Обновляет субтитры
- Проверяет границы сегмента для автоостановки (с точностью 20 мс)
- И только после всех проверок обновляет индекс текущего предложения

При остановке или паузе цикл отменяется через cleanup.

### Разбор функции `updateProgress`

#### 1. Базовая инициализация

Внутри эффекта, мы инициализируем функцию `updateProgress`. В ней мы:
- Получаем текущее время
- Форматируем его для UI
- Обновляем субтитры
- Забираем конечный результат функции `getCurrentSegmentBounds()`

#### 2. Проверка границ сегмента

После чего, мы проверяем:
- Если включён режим Sentence Mode и есть маркеры, то переходим в следующее условие
- Если текущее время аудио больше или равен конечному результату, то мы останавливаем воспроизведение

#### 3. Основная логика остановки

```javascript
if (isPlayMode && timeMarkers?.length) {
  const { end } = getCurrentSegmentBounds();
  //      ^^^ Конец текущего сегмента (например, 10.0s)

  // Проверка с точностью 20 мс до конца
  if (currentTimeValue >= end - 0.02) {
    instance.pause();  // Остановить воспроизведение
    
    // Запланировать ещё ОДИН кадр для финальной отрисовки
    if (isPlaying) {
      animationFrame = requestAnimationFrame(updateProgress);
    }
    return;  // ВЫХОД из функции - НЕ обновляем маркер!
  }
}
```

#### Почему нужно выходить из функции но не обновлять маркер?

**БЕЗ return:**
```
─────────────────────────────▶ currentTime = 9.985
                               end = 10.0
1. instance.pause() ✅
2. updateCurrentMarkerIndex(9.985) ❌ ПЛОХО!
   → может перепрыгнуть на следующий маркер
3. RAF продолжается
```

**С return:**
```
─────────────────────────────▶ currentTime = 9.985
1. instance.pause() ✅
2. if (isPlaying) RAF → для обновления UI ✅
3. ВЫХОД → маркер НЕ обновляется ✅
```

### Роль `updateCurrentMarkerIndex(currentTimeValue)`

Эта функция:
- Находит, между какими маркерами находится `currentTimeValue`
- Обновляет `currentMarkerIndex` в Redux
- Вызывается только если мы НЕ на границе сегмента

### Планирование следующего кадра

```javascript
if (isPlaying) {
  animationFrame = requestAnimationFrame(updateProgress);
}
```

**Визуализация цикла:**

```
Кадр 1 → updateProgress() → if (isPlaying) → запланировать Кадр 2

Кадр 2 → updateProgress() → if (isPlaying) → запланировать Кадр 3

Кадр 3 → updateProgress() → if (isPlaying) → запланировать Кадр 4
...

[ПАУЗА - isPlaying становится false]

Кадр N → updateProgress() → if (isPlaying) → НЕТ → цикл СТОП
```

**При нажатии Play:**
1. `isPlaying` меняется на `true`
2. `useEffect` перезапускается
3. Запускается первый RAF-кадр
4. Начинается цикл

### Остановка цикла - cleanup функция

```javascript
return () => {
  if (animationFrame) {
    cancelAnimationFrame(animationFrame);
  }
};
```

**Вызывается:**
- При размонтировании компонента
- При изменении любой зависимости (перед перезапуском эффекта)
- При паузе (`isPlaying` становится `false`)

### Пример полного цикла воспроизведения

```
Сегмент: [5.0s ─────────────► 10.0s]

Воспроизведение:
  Frame 1:    5.016s → updateProgress() → всё норм → запланировать Frame 2
  Frame 2:    5.032s → updateProgress() → всё норм → запланировать Frame 3
  ...
  Frame N:    9.967s → updateProgress() → всё норм → запланировать Frame N+1
  Frame N+1:  9.983s → currentTime >= 10.0 - 0.02? ✅ ДА!
                    → instance.pause()
                    → return (БЕЗ updateCurrentMarkerIndex!)
                    → RAF завершается
```

### ✅ Правильный порядок операций:

1. **Обновить UI времени** ← визуальная обратная связь
2. **Обновить субтитры** ← синхронизация текста
3. **Проверить границы сегмента** ← логика остановки
4. **Обновить индекс маркера** ← ТОЛЬКО если не останавливаемся
5. **Запланировать следующий кадр**

## 8. Зачем буфер в 20 миллисекунд?

```javascript
if (currentTimeValue >= end - 0.02)
```

### Без буфера:

```
end = 10.0s
Frame N:   9.983s → 9.983 >= 10.0? ❌ продолжаем
Frame N+1: 10.017s → 10.017 >= 10.0? ✅ стоп
           ^^^^^^
           Уже ПРОСРОЧИЛИ на 17 мс!
```

### С буфером:

```
end = 10.0s 
Frame N: 9.983s → 9.983 >= 9.98? ✅ стоп заранее
        ^^^^^
        Остановились за 17 мс до конца - идеально!
```

### Почему именно 20 мс:

- RAF ~16.67 мс между кадрами (60 FPS)
- Даёт запас на один кадр
- Более точная остановка, чем раньше (где было 0.1 секунды)

## 9. Полный цикл жизни RAF

```javascript
// 1. Пользователь нажимает Play
dispatch(setIsPlaying(true));

// 2. isPlaying меняется → useEffect запускается
useEffect(() => {
  if (isPlaying) {
    animationFrame = requestAnimationFrame(updateProgress);
    // Запущен RAF-цикл
  }
}, [isPlaying, ...]);

// 3. Цикл работает ~60 раз/сек
updateProgress() → updateProgress() → updateProgress() → ...

// 4. Достигли конца сегмента
currentTimeValue = 9.985s → pause() → return

// 5. Пользователь нажимает Play снова
dispatch(setIsPlaying(true));

// 6. useEffect cleanup вызывается:
return () => {
  cancelAnimationFrame(animationFrame); // Отменить старый
};

// 7. useEffect запускается заново → новый RAF-цикл
```
